# Public Demo Mode Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Remove login from the public Vercel experience and keep all SpeakCoach data only until the next browser refresh.

**Architecture:** A build-time `NEXT_PUBLIC_DEMO_MODE` flag selects an in-memory storage adapter and a fixed demo user. Client initialization clears legacy `speakcoach_*` browser data before rendering, while the Next.js proxy redirects `/login` and expires old authentication cookies.

**Tech Stack:** Next.js 16.2.10, React 19.2.4, TypeScript 5, Zustand 5.0.14, Node.js 24 test runner, Vercel

## Global Constraints

- Production uses `NEXT_PUBLIC_DEMO_MODE=true`; other environments default to the existing login flow.
- Only keys beginning with `speakcoach_` may be deleted from browser storage.
- Demo records, reports, and uploaded text remain available during client navigation and disappear after a full refresh.
- Existing authentication API routes remain in the repository.
- No phone number, authentication cookie, or persistent user data is required in demo mode.

- [ ] **Step 3: Implement the demo-mode module and storage adapter**

Create `src/lib/demo-mode.ts` with a fixed `DEMO_USER`, a `Storage`-compatible in-memory implementation, protected cleanup for `speakcoach_*`, and helpers that route application reads and writes to memory when `NEXT_PUBLIC_DEMO_MODE` is exactly `true`.

```ts
import type { AppUser } from "@/types";

export type StorageLike = Pick<Storage, "length" | "key" | "getItem" | "setItem" | "removeItem" | "clear">;
export const isDemoMode = (value = process.env.NEXT_PUBLIC_DEMO_MODE) => value === "true";
export const DEMO_MODE = isDemoMode();
export const DEMO_USER: AppUser = { id: "demo-user", displayName: "演示访客", phone: "" };

export function createMemoryStorage(): StorageLike {
  const values = new Map<string, string>();
  return {
    get length() { return values.size; },
    key(index) { return [...values.keys()][index] ?? null; },
    getItem(key) { return values.get(key) ?? null; },
    setItem(key, value) { values.set(key, value); },
    removeItem(key) { values.delete(key); },
    clear() { values.clear(); },
  };
}
```

The same module owns one memory store, clears only collected prefix-matching keys, protects browser storage access with `try/catch`, and exposes application read/write/remove helpers plus an idempotent `initializeDemoSession()`.

Update `src/lib/mock-api.ts` so user, record, report, and uploaded-file persistence uses the new storage helpers. In demo mode `getCurrentUser()` returns `DEMO_USER` and no phone number is stored.

- [ ] **Step 2: Gate child rendering until cleanup completes**

Update `ClientInit` so demo mode calls `initializeDemoSession()`, resets Zustand once, and installs `DEMO_USER`. The empty in-memory adapter prevents old data from being read on the initial render, while normal mode continues to call `initUser()` without clearing data.

```tsx
export function ClientInit({ children }: { children: React.ReactNode }) {
  const { initUser, resetSession, setUser } = useAppStore();
  useEffect(() => {
    if (DEMO_MODE) {
      if (initializeDemoSession()) resetSession();
      setUser(DEMO_USER);
      return;
    }
    initUser();
  }, [initUser, resetSession, setUser]);
  return <>{children}</>;
}
```
